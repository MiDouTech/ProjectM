# AI 开发协作规则

> 本 skill 是在本仓库工作的协作约束总纲。与 `CLAUDE.md` 冲突时按「指令优先级」处理。

## 一、语言规范
1. 所有 Git commit message 必须使用中文。
2. 所有 PR 相关内容，包括标题、描述、Review 说明和回复等，必须使用中文。
3. 过程反馈与结果说明尽量使用中文；专业术语、技术名词、代码标识及产品专有名称保留英文。

## 二、指令优先级
4. 当规则发生冲突时，按以下优先级执行：用户当前明确指令 > 项目级 `AGENTS.md`/`CLAUDE.md` > 仓库文档与既有约定 > 通用规则。
5. 发现指令之间存在冲突时，必须主动说明冲突点及建议处理方式，不得静默选择或自行忽略其中一项。
6. 优先遵循仓库现有架构、代码风格、测试方式、命名规范和工具链，不得仅凭个人偏好替换既有成熟实现。

## 三、任务理解与方案审核
7. 开始编码前，必须先向用户说明：对任务的理解；用户最想解决的核心问题；当前仍存在的歧义；直接开始实现最可能产生误解的地方；拟定的执行计划。
8. 开发前必须将需求转换为可验证的验收条件。
9. 如果验收标准缺失，或者存在多种会显著影响结果的合理实现，应先说明当前假设、实现差异及建议方案，等待用户确认。
10. 当用户要求"出方案"时，只提供方案，不得开始开发；必须等待用户审核并明确同意后，才能进入开发阶段。
11. 若用户已经明确批准方案或明确要求按照既定方案直接执行，则无需重复请求相同授权，但仍须说明任务理解和执行计划。
12. 不得把推测描述成事实；关键判断应明确区分：已验证事实；根据现有信息作出的推断；尚未确认的假设。
13. 信息不足时，优先通过只读检索进行验证；如果不同假设会导致明显不同的实现结果，必须先向用户确认。

## 四、检索与影响范围评估
14. 修改任何内容前，必须优先使用 Grep/Glob 检索相关代码、配置、测试、引用和文档，评估影响范围。
15. 影响范围评估至少应关注：定义位置；调用或引用位置；相关测试；配置与环境变量；公共接口；文档；潜在兼容性影响。
16. 对大文件优先使用 Grep 定位目标内容，避免直接完整 Read；确需读取时，必须使用 offset/limit 分段读取。
17. 当预计 Grep 命中较多时，先使用 `output_mode="count"` 统计命中数量，再根据结果决定检索和读取策略。
18. 遇到单次上下文过大或任务规模较大时，必须主动拆分任务。
19. 长任务每读取 5～10 个文件，应进行一次阶段性小结，记录：已确认的信息；尚未确认的问题；当前影响范围；下一步检索或开发方向。
20. 阶段性小结应提炼有效结论，不得重复堆积原始文件内容，避免上下文持续膨胀或溢出。

## 五、工作区与改动范围控制
21. 修改前必须检查工作区状态，识别用户已有的未提交改动、新增文件和当前分支状态。
22. 不得覆盖、撤销、暂存、提交或清理不属于当前任务的用户改动。
23. 只修改完成当前任务所必需的内容，不得未经授权：顺手重构无关代码；格式化无关文件；升级无关依赖；修改无关配置；改变任务范围外的现有行为。
24. 发现关联问题但不影响当前任务时，只记录并反馈，不得擅自扩大修改范围。
25. 开发过程中，如果发现实际工作量、风险或影响范围明显超出原计划，应暂停扩展，说明变化、原因及影响，并重新确认计划。

## 六、高风险操作与授权
26. 执行下列高风险或不可逆操作前，必须说明影响范围、风险和恢复方式，并取得用户明确同意：删除或覆盖文件；重置、清理或回退代码；修改或删除数据；执行数据库迁移；强制推送；修改生产环境；修改线上配置；执行可能造成服务中断的操作。
27. 未经用户明确授权，不得执行以下外部状态变更：合并 PR；发布版本；部署服务；修改远程仓库状态。
28. 用户明确要求"提交""推送""创建 PR"或批准包含这些操作的执行计划，即视为对应操作已获授权；授权不得自动扩大到未提及的其他操作。

## 七、依赖、架构与兼容性
29. 引入新依赖前，必须说明：引入的必要性；现有方案为何不足；可选替代方案；包体积、性能、安全和维护成本；对现有环境的影响。
30. 调整公共 API、公共组件、数据结构、数据库结构或整体架构前，必须说明影响范围、兼容性和迁移成本，并取得用户确认。
31. 修改公共行为时，必须检查：向后兼容性；配置兼容性；数据兼容性；调用方兼容性；部署与回滚兼容性。
32. 存在破坏性变化时，必须明确标注，不得把 Breaking Change 当作普通内部修改处理。
33. 中高风险改动必须提供回滚方案，至少说明如何撤销代码、配置和数据变更。

## 八、开发优先级与提交策略
34. 大型任务必须按照 P0、P1、P2 划分优先级，并按优先级顺序开发：P0 阻塞核心功能、安全或关键流程的问题；P1 重要功能、主要缺陷或明显影响体验的问题；P2 优化、增强、低风险改进或非阻塞事项。
35. 不得在 P0 尚未完成并验证时，擅自开发与其无关的 P1 或 P2 内容。
36. 不同优先级的改动必须分开提交，不得将 P0、P1、P2 混入同一个 commit。
37. 每个优先级可以包含一个或多个原子 commit，不强制把同一优先级的所有改动压缩成一个 commit。
38. 每个 commit 必须：保持单一目的；可独立 Review；尽可能可独立验证；尽可能可独立回滚；不包含无关格式化或顺手重构。
39. 每完成一个 commit，必须对该 commit 的 diff 进行 Review。
40. Review 发现问题时，应集中一次性修正同类问题并重新检查，确认无误后再向用户反馈，避免逐个零散修复。

## 九、测试与验证
41. 编码前应根据任务风险确定测试范围，包括：正常路径；边界条件；异常路径；权限与安全场景；兼容性；回归风险。
42. 修复 Bug 时，条件允许应先补充能够稳定复现问题的测试，再实施修复。
43. 开发完成后，必须运行与改动直接相关的：自动化测试；静态检查；类型检查；Lint；构建检查；必要的手动验收。
44. 测试失败时，必须先判断失败是否由当前改动引起，不得为了让测试通过而擅自：删除测试；跳过测试；弱化断言；扩大容错范围；隐藏错误；修改与任务无关的测试配置。
45. 无法执行某项验证时，必须明确说明：无法执行的原因；未验证的范围；可能存在的风险；建议的后续验证方式。
46. 未实际运行或确认的测试，不得描述为"已通过"。
47. 所有"已完成""已修复""测试通过""没有问题"等结论，必须附带可核查依据，包括执行的命令、关键结果和未覆盖范围。

## 十、自检与提交前检查
48. 开发完成后，必须先进行全面自检，确认实现、测试和改动内容无误后，才能提交。
49. 自检至少包括：实现是否符合需求和验收条件；是否遗漏调用方、测试、配置或文档；是否引入兼容性问题；是否存在重复、无效或调试代码；是否包含无关修改；是否存在明显安全或性能风险。
50. 提交前必须检查完整 diff 和暂存区，确认不存在：调试代码；临时文件；无意义日志；构建产物；敏感信息；意外格式化；用户的无关改动。
51. 自检发现问题时，应先集中修复并重新验证，不得在已知存在问题的情况下提交或声称任务完成。

## 十一、安全与隐私
52. 不得读取、输出、提交或记录密钥、Token、密码、Cookie、私钥及个人敏感信息。
53. 检索和展示配置文件、日志或环境变量时，应主动避免暴露敏感值；必要时只展示字段名或脱敏内容。
54. 发现疑似凭证或敏感信息时，应立即停止传播，说明发现位置和风险，并建议用户撤销、轮换或清理相关凭证。
55. 不得将密钥、凭证或敏感隐私信息写入代码、文档、日志、测试数据、commit message、PR 描述或持久记忆。

## 十二、文档同步
56. 当改动影响以下内容时，必须同步更新对应文档：公共接口；配置项；环境变量；部署方式；使用流程；数据结构；开发约定；用户可观察行为。
57. 纯内部实现调整且不影响使用方式时，无需为了形式完整而机械新增或修改文档。
58. 文档内容必须与实际实现一致，不得描述尚未实现或未经验证的能力。

## 十三、最终交付
59. 开发完成后的最终答复必须提供简明总结清单，包括：改动范围；风险；测试；验收；后续事项。
60. 总结清单必须简明扼要，能用一句话说明的内容不得拆成两句话。
61. 最终答复必须提供用户能够复现的验收步骤；涉及环境依赖时，应注明必要的前置条件。
62. 如果仍有未完成、未验证或需要用户决定的事项，必须明确列出，不得隐藏在"已完成"的结论中。
63. 最终交付应明确区分：已完成；已验证；未验证；已知风险；建议后续处理。

## 十四、每次回答的工作总结
64. 每次回答末尾必须附上一段不超过 200 字的中文"工作总结"。
65. 工作总结应如实说明 AI 本次完成的操作，例如：读取了多少文件；执行了哪些检索；修改了哪些内容；运行了哪些测试；完成了哪些验证；是否执行了提交或外部操作。
66. 没有执行文件、代码或工具操作时，也必须如实说明，不得虚构工作过程。
67. 工作总结不得超过 200 字，且必须放在回答的最后。

## 十五、规则适用边界
68. 对纯咨询、翻译、文字润色、内容整理等不涉及代码或文件修改的任务，可以跳过 Git、测试、提交和代码检索相关流程。
69. 非开发任务仍须遵守语言规范、任务理解、不确定性说明、安全隐私及回答末尾工作总结规则。
70. 对简单、明确、低风险的任务，可以压缩任务理解和执行计划，但不得省略会影响结果的关键歧义或风险。
71. 流程规则的目的在于降低误解和交付风险，不得为了机械遵守流程而制造无意义步骤；任何简化都不得牺牲安全性、正确性和可验证性。
