# NPSS 算分与硬校验规则（事实源）

> 来源：《项目管理规范（NPSS）》《干系人》。本文件是验收域、奖金、立项审批 guard 的唯一实现依据。AI 据此写 Service 与测试。

## 1. NPSS 是什么（实现者必读）
NPSS（Net Project Success Score，净项目成功得分）= 交付后 6–12 个月，由**加权干系人**对"交付价值是否值得投入"打 0-10 分得出的价值验收标准。**不替代铁三角，而是更高层价值衡量。** 项目验收两段：①结果验收（时间-成本-范围铁三角，达标即可结案）；②价值验收（NPSS，延后触发）。

## 2. 评价时点（定时任务触发）
- 默认：项目**结案后 6–12 个月**自动发起 NPSS 评分（`pm_project.value_review_due_date`，建议默认 = 结案日 + 9 个月，可由 PMO 调）。
- 长周期项目：可在每个主要里程碑后做阶段性 NPSS（`pm_npss_review.round`）。
- 实现：定时 Job 扫描 `value_review_due_date <= today 且 status=已结案` 的项目 → 置 `价值验收中` → 建 `pm_npss_review(status=pending)` → 给每个干系人建评分待办 → 经 MessageProvider 通知。

## 3. 单项目算分
> 评分单位有两种口径，落地以**评价主体**口径为准；个人口径为其退化形态，保留向后兼容。

### 3A. 评价主体口径（V48，主口径）
1. NPSS 计算方式 = **评价主体 + 评价规则**。评价主体（`pm_npss_subject`）= 名称 + 权重占比（启用主体合计 100%）+ 是否受益方；每个主体可挂**多个成员（成员即干系人）**。评价规则固定为"各主体加权求和"。
2. 每个成员（干系人）打 0-10 分。**同一主体多人 → 先取这些人分数的平均，再乘主体权重**。
3. 加权满意度（0-100）= (Σ(weight_j × avg_j) / Σweight_j) × 10，其中 avg_j = 主体 j 成员评分平均；无成员评分的主体跳过，其权重不参与归一化。
   - 例：业务方(权重30) 成员[9,10]→平均9.5；团队(权重70) 成员[8] → (30×9.5 + 70×8)/100 ×10 = 84.5。
4. 配置层级：**租户级模板（`pm_npss_subject_template`）→ 项目可覆盖**；发起评价时物化为项目级实例。
5. 并签：所有主体的所有成员全部打分完成后才汇总（`NpssReviewService.allScored`）。

### 3B. 个人口径（兼容，退化形态）
1. 每个干系人打 0-10 分（`pm_npss_score.score`）。
2. 加权满意度（0-100）= Σ(score_i × weight_i) × 10，其中 Σweight_i = 1（权重以百分比存，计算前归一化）。
   - 例：发起人9×0.3 + 业务10×0.3 + 团队8×0.1 + 财务9×0.1 + 其他9×0.2 = 9.3 → ×10 = 93。
3. 结果分级（`result_level`）：
   - **Success**：加权满意度对应原始均分 9–10（即 weighted_score ≥ 90）
   - **Mixed**：7–8（70 ≤ weighted_score < 90）
   - **Failure**：0–6（weighted_score < 70）
4. 鼓励真实反馈：`comment` 必填提示，禁"人情分"。结果用于组织学习，非惩罚团队。

## 4. 权重规则（立项阶段对齐，硬校验）
- 权重是**项目级**的，立项阶段由 Leader 与干系人对齐。按项目类型预置默认模板（见第 6 节），可微调。
- **硬校验 A（受益方权重）**：受益方（sponsor 发起人 + business 业务方）权重之和 ≥ 其他所有干系人权重之和的 **2 倍** 不是要求；准确规则为：受益方单类权重 ≥ 其他单类权重的约 2 倍，落地实现取"受益方合计 ≥ 50%"。保存权重时校验，不满足报错。
- **硬校验 B（权重和）**：所有干系人权重之和必须 = 100%（允许 ±0.01 浮点误差），否则不可提交立项。
- **主体口径（V48）**：上述 A/B 同样作用于评价主体——启用主体权重合计=100%、受益方主体（`beneficiary=1`）合计≥50%，由 `SubjectWeightValidator` 校验；个人口径校验由 `WeightValidator` 保留兼容。

## 5. PMO 总体评价
PMO NPSS = 成功项目占比(%) − 失败项目占比(%)。
- 例：100 个项目，60 成功、10 失败、30 混合 → PMO NPSS = 60 − 10 = 50。
- 全球基线 36（48% 成功 − 12% 失败）。**米多目标：PMO NPSS > 36**。
- 实现：报表域按财年聚合 `result_level` 计算。

## 6. 项目类型 → 默认权重模板
| 类型 | 默认权重 |
|---|---|
| 战略级 S | 发起人30 / 业务方30 / 团队10 / 财务10 / 其他20(含管理层) |
| 创新级 I | 同 S，可调（POC 侧重业务方/发起人） |
| 运营级 O·常规 | 业务使用部门50 / 管理层20 / 团队10 / 财务10 / 其他10 |
| 运营级 O·定向整改 | 同内部业务模板 |
| 运营级 O·专项督办 | 监管/审计30 / 管理层30 / 业务20 / 团队10 / 其他10 |

（行业通用参考：外部客户项目 客户40/业务对接20/管理10/团队10/财务10/其他10。）

## 7. 奖金硬校验（与 NPSS 联动）
项目奖金总额 = 项目类型奖金基数 × 项目价值系数(基准1) × (干系人满意度得分 / 100)。
- **硬规则 1（归零线）**：满意度 < 60 → 奖金总额**直接归零**（所有类型）。
- **硬规则 2（无奖金类型）**：O·定向整改、O·专项督办 **无项目奖金**，不参与奖金计算。
- 奖金基数与价值系数由 PMO 另行发布，系统留配置项。

## 8. 立项职级 guard（审批节点强校验）
- 门槛由项目类型属性 `pm_project_type.min_job_level` 决定（取代原按 S/I/O 硬编码），空=不限。
- 内置种子默认：战略级 S → **L3+**；运营级 O → **L2+**；创新级 I → 不限。租户可改。
- 实现：门槛在提交立项时随上下文（`minJobLevel`）传入审批引擎，节点 guard `JOB_LEVEL` 读 `sys_user.job_level` 与门槛比较；项目状态机注册 guard 共用 `JobLevelRule`（门槛制，单一事实源），不满足直接 reject 并提示。

## 9. 必备单元测试清单（AI 必须生成）
1. 加权满意度计算正确（含归一化）。
2. result_level 三档边界（70、90 临界）。
3. 受益方权重合计 < 50% 时保存被拒。
4. 权重和 ≠ 100% 时提交立项被拒。
5. 满意度 59 → 奖金 0；60 → 正常。
6. O·定向整改/专项督办 → 奖金计算返回 0 且不报错。
7. S 类 Leader 为 L2 → 立项审批被拒；L3 → 通过 guard。
